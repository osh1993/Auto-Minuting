# NotebookLM MCP 서버 검증

## MCP 서버 상태

프로젝트에 `notebooklm-mcp` 서버가 설정되어 있다.

### 사용 가능한 도구 목록

| 도구 | 설명 | 용도 |
|------|------|------|
| `source_add` | 소스 추가 (text, url, file, drive) | 전사 텍스트를 노트북에 소스로 등록 |
| `note_create` | 노트 생성 | 회의록 생성 요청 |
| `note_list` | 노트 목록 조회 | 생성된 회의록 확인 |
| `note_update` | 노트 수정 | 회의록 수정 |
| `note_delete` | 노트 삭제 | 정리 |
| `studio_create` | 스튜디오 콘텐츠 생성 (audio, video 등) | 오디오 요약 등 |
| `download_artifact` | 콘텐츠 다운로드 | 생성 결과물 다운로드 |

### 인증 방법

```bash
nlm login           # 자동 인증
nlm login switch <profile>  # 계정 전환
```

## 테스트 시나리오

### 시나리오 1: 새 노트북에 전사 텍스트 추가 후 회의록 생성

1. NotebookLM에서 새 노트북 생성 (수동 또는 MCP 통해)
2. `source_add(source_type="text", text=전사텍스트)` - 전사 텍스트를 소스로 추가
3. `note_create(content="이 소스의 내용을 바탕으로 구조화된 회의록을 작성해주세요...")` - 회의록 생성 요청
4. `note_list()` - 생성된 노트 확인
5. 생성된 회의록의 품질 평가

### 시나리오 2: 기존 노트북에 소스 추가 후 누적 회의록

1. 기존 프로젝트 노트북 선택
2. `source_add(source_type="text", text=새_전사텍스트)` - 추가 회의 전사 텍스트 등록
3. `note_create(content="최근 추가된 소스를 기반으로 회의록을 작성해주세요")` - 회의록 생성
4. 이전 회의 소스와의 연관성이 반영되는지 확인

## 테스트 결과

**상태:** 테스트 대기 (MCP 서버 실행 환경 필요)

MCP 서버는 프로젝트에 설정되어 있으나, 실제 도구 호출 테스트는 MCP 연결이 활성화된 환경에서 진행해야 한다.

### 예상 평가 기준

| 기준 | 설명 |
|------|------|
| source_add 성공 | 텍스트 소스가 정상 등록되는가 |
| note_create 성공 | 회의록 생성 요청이 처리되는가 |
| 회의록 품질 | NotebookLM이 소스 기반 회의록을 생성하는가 |
| 응답 시간 | 회의록 생성까지 걸리는 시간 |
| 안정성 | 반복 호출 시 일관된 결과를 제공하는가 |

## 제약사항

### PC 의존성

- MCP 서버는 로컬 PC에서 실행되어야 한다
- Android 앱에서 직접 MCP 서버를 호출할 수 없다
- **모바일 독립 실행 불가** - PC가 중계 역할을 해야 함
- 사용자가 회의 후 PC 앞에 있어야 자동화 파이프라인이 동작

### Google 계정 인증

- `nlm login`으로 Google 계정 인증 필요
- 인증 토큰 만료 시 재인증 필요
- 멀티 계정 환경에서 프로파일 관리 필요

### 안정성 리스크

- **비공식 경로**: Google이 NotebookLM UI를 변경하면 MCP 서버가 동작하지 않을 수 있음
- **버전 호환성**: MCP 서버 업데이트가 NotebookLM 변경을 추적해야 함
- **Rate Limiting**: NotebookLM의 사용량 제한이 불명확
- **에러 처리**: 비공식 API이므로 에러 코드/메시지가 체계적이지 않을 수 있음

### 기능 범위

- NotebookLM의 강점: 소스 관리, 오디오 오버뷰 생성, 멀티소스 기반 분석
- 회의록 생성은 NotebookLM의 핵심 기능은 아님 (노트 기능으로 우회)
- 프롬프트 기반 회의록 생성의 세밀한 제어가 제한적

## Gemini API vs MCP 비교

| 기준 | Gemini API 직접 호출 | NotebookLM MCP 서버 |
|------|---------------------|---------------------|
| **모바일 독립성** | 가능 (Firebase AI Logic SDK) | 불가 (PC 필요) |
| **안정성** | 높음 (공식 API) | 낮음 (비공식 경로) |
| **프롬프트 제어** | 완전 제어 | 제한적 (NotebookLM 경유) |
| **비용** | 무료 티어 존재 | 무료 (NotebookLM 무료) |
| **설정 복잡도** | 낮음 (API 키만 필요) | 높음 (MCP 서버 + 인증) |
| **오디오 직접 입력** | 가능 (최대 9.5시간) | 불가 |
| **소스 관리** | 없음 (직접 구현 필요) | 있음 (NotebookLM 내장) |
| **멀티소스 분석** | 없음 | 있음 (여러 회의 교차 분석) |
| **프로덕션 적합성** | 높음 | 낮음 (PC 의존 + 비공식) |
| **유지보수 부담** | 낮음 (Google 공식 지원) | 높음 (UI 변경 추적 필요) |

### 비교 결론

**1차 채택 권장: Gemini API 직접 호출**
- 모바일 독립 실행 가능 (프로젝트 핵심 요구사항 충족)
- 공식 API로 안정성 보장
- 프롬프트 완전 제어 가능
- 오디오 직접 입력이라는 대안 파이프라인 가능성

**2차 폴백: NotebookLM MCP 서버**
- PC 연결 시 소스 관리/멀티소스 분석 기능 활용 가능
- Gemini API 불가 상황에서의 대안
- 장기적으로 Enterprise API 공개 시 전환 가능성

## 평가: LIMITED

**판정 근거:**

| 기준 | 결과 |
|------|------|
| MCP 서버 설정 | 확인 (프로젝트에 설정됨) |
| 도구 사용 가능 | 확인 (source_add, note_create 등) |
| 모바일 독립 실행 | 불가 (PC 필요) |
| 안정성 | 낮음 (비공식 경로) |
| 프로덕션 적합성 | 낮음 |
| 실제 테스트 | 미완료 (MCP 환경 필요) |

**LIMITED** - 기능적으로 동작 가능하나, PC 의존성과 비공식 경로의 안정성 리스크로 인해 1차 채택 경로로는 부적합. 폴백 또는 보조 도구로 활용 가능.
