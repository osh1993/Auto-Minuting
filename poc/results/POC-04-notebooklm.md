# POC-04: 회의록 생성 경로 검증 결과

## 검증 요약

전사 텍스트를 입력으로 AI가 구조화된 회의록을 생성하는 경로가 확인되었다. **Gemini API 직접 호출이 1차 채택 경로**이며, NotebookLM MCP 서버는 폴백으로 활용 가능하다.

## 경로별 결과

### 경로 1: Gemini API 직접 호출

| 항목 | 결과 |
|------|------|
| **테스트 방법** | google-genai SDK, gemini-2.5-flash 모델 |
| **프롬프트 설계** | 역할 지정 + 4섹션 구조화 출력 + 작성 지침 |
| **모바일 독립 실행** | 가능 (Firebase AI Logic SDK 경유) |
| **비용** | 무료 티어 내 운용 가능 (일일 500회 제한, 실사용 5회 미만) |
| **안정성** | 높음 (Google 공식 API) |
| **판정** | NEEDS_API_KEY (인프라 준비 완료, 키 발급 후 품질 검증 필요) |

**상세 결과:** `poc/minutes-test/gemini-api-test.md` 참조

- 테스트 스크립트 (`gemini-test.py`) 작성 완료
- 프롬프트 설계 완료 (회의 개요, 안건, 결정 사항, 액션 아이템 4섹션)
- Android 통합 코드 예시 (Firebase AI Logic SDK, Kotlin) 작성
- API 키 발급 후 `GEMINI_API_KEY=xxx python gemini-test.py`로 실행 가능

### 경로 2: NotebookLM MCP 서버

| 항목 | 결과 |
|------|------|
| **MCP 서버 상태** | 프로젝트에 설정 확인 |
| **사용 가능 도구** | source_add, note_create, note_list 등 |
| **모바일 독립 실행** | 불가 (PC에서 MCP 서버 실행 필요) |
| **안정성** | 낮음 (비공식 경로, Google UI 변경 시 깨질 수 있음) |
| **판정** | LIMITED (기능 동작 가능하나 프로덕션 부적합) |

**상세 결과:** `poc/minutes-test/notebooklm-mcp-test.md` 참조

- PC 의존성이 "원클릭 모바일 파이프라인" 목표와 충돌
- 비공식 경로로 장기 유지보수 리스크 존재
- 다만 소스 관리/멀티소스 분석 등 고유 기능 존재

### 경로 3: NotebookLM Enterprise API

| 항목 | 결과 |
|------|------|
| **상태** | Alpha (2025년 9월 출시) |
| **접근성** | 기업 전용 (Cloud NotebookLM User IAM, Google Cloud 프로젝트 필요) |

### 경로 4: Gemini API 오디오 직접 입력 (대안 파이프라인)

| 항목 | 결과 |
|------|------|
| **개념** | STT 단계를 건너뛰고 오디오 파일을 Gemini에 직접 입력 |
| **지원** | Gemini 2.5 Flash - 최대 9.5시간 오디오 처리 가능 |
| **장점** | 파이프라인 단순화 (녹음 -> 회의록, STT 생략) |
| **단점** | Galaxy AI 전사의 한국어 정확도 대비 품질 미검증 |
| **판정** | 대안 파이프라인으로 기록 (Phase 5에서 비교 검증 예정) |

오디오 직접 입력 파이프라인은 STT 단계를 완전히 생략할 수 있어, Galaxy AI 전사가 불가능한 상황에서의 폴백으로도 활용 가능하다.

## Go/No-Go

### Go

**근거:**
1. Gemini API 직접 호출 경로가 기술적으로 실현 가능 (공식 API, SDK, 무료 티어)
2. 모바일 독립 실행 가능 (Firebase AI Logic SDK로 Android 네이티브 통합)
3. 프롬프트 기반 구조화된 회의록 생성이 가능한 인프라 확인
4. 폴백 경로 (MCP 서버) 및 대안 파이프라인 (오디오 직접 입력) 존재

**리스크:**
- 실제 API 호출 테스트 미완료 (API 키 필요) - 다만 Gemini API는 공식 서비스이므로 동작 실패 리스크 극히 낮음
- 한국어 회의록 품질은 프롬프트 튜닝으로 개선 가능

## 채택 경로

### 1차: Gemini API 직접 호출 (Firebase AI Logic SDK 경유)

- **방식:** Firebase AI Logic SDK -> Gemini 2.5 Flash
- **입력:** 전사 텍스트 (Galaxy AI STT 출력)
- **출력:** 구조화된 Markdown 회의록
- **모바일 독립 실행:** 가능
- **비용:** 무료 티어 내 운용

### 2차 (폴백): NotebookLM MCP 서버

- **방식:** PC에서 MCP 서버 -> source_add + note_create
- **조건:** PC 연결 가능한 환경, Gemini API 불가 시
- **장점:** 소스 관리, 멀티소스 분석 가능
- **단점:** PC 의존, 비공식 경로

### 대안 파이프라인: Gemini API 오디오 직접 입력

- **방식:** 녹음 파일을 Gemini API에 직접 전달 (STT 건너뛰기)
- **조건:** Galaxy AI STT가 불가능하거나 정확도가 낮은 경우
- **장점:** 파이프라인 단순화, STT 의존성 제거
- **단점:** 품질 미검증, 오디오 파일 크기/처리 시간

## 프롬프트 전략

### 효과적인 회의록 생성 프롬프트 구조

```
1. 역할 부여: "당신은 전문 회의록 작성자입니다"
2. 구조화된 출력 형식 지시:
   - 회의 개요 (날짜, 참석자, 시간)
   - 주요 안건 및 논의 내용 (안건별 구분)
   - 결정 사항 (번호 매김)
   - 액션 아이템 (테이블: 담당자 | 할 일 | 기한)
3. 작성 지침:
   - 핵심 내용만 간결하게 요약
   - 실제 이름 사용
   - 결정/액션 빠짐없이 포함
   - 한국어 작성
4. 입력: 전사 텍스트 전문
```

### 프롬프트 최적화 방향

- **형식 커스터마이징:** 사용자가 회의록 형식을 선택할 수 있도록 프롬프트 템플릿 관리
- **컨텍스트 추가:** 회의 종류(정기회의/브레인스토밍/의사결정)에 따른 프롬프트 분기
- **품질 개선:** few-shot 예시 추가, 출력 형식 더 세밀하게 지정
