2. **테스트용 샘플 전사 텍스트 준비** (sample-transcript.txt):
   한국어 회의 전사 샘플을 작성한다 (최소 20줄). 다음 요소 포함:
   - 3명 이상의 화자 (화자 A, 화자 B, 화자 C)
   - 회의 주제: 프로젝트 진행 상황 보고
   - 논의 내용: 이슈 리포트, 일정 확인, 결정 사항
   - 액션 아이템: 담당자와 기한이 포함된 할 일
   - 자연스러운 한국어 대화체 (실제 회의 느낌)

3. **Gemini API 직접 호출 테스트** (gemini-api-test.md, gemini-test.py):
   a) Google AI Studio에서 API 키 발급 방법 문서화 (https://aistudio.google.com/apikey)
   b) Python 테스트 스크립트 작성 (gemini-test.py):
      - google-genai 패키지 사용: `pip install google-genai`
      - 환경변수: GEMINI_API_KEY
      - 모델: gemini-2.5-flash
      - 프롬프트: 전사 텍스트를 입력으로 구조화된 회의록 생성
        - 프롬프트에 포함할 형식 지시:
          1. 회의 개요 (날짜, 참석자)
          2. 주요 안건 및 논의 내용
          3. 결정 사항
          4. 액션 아이템 (담당자, 기한)
      - sample-transcript.txt를 읽어서 프롬프트에 삽입
      - 결과를 output-minutes.md로 저장
      - 사용법 주석: `# 사용법: GEMINI_API_KEY=xxx python gemini-test.py`
   c) gemini-api-test.md에 기록:
      - ## API 설정 (키 발급 방법, Firebase AI Logic SDK vs google-genai 비교)
      - ## 프롬프트 설계 (사용한 프롬프트, 형식 지시)
      - ## 테스트 결과 (생성된 회의록 품질 평가 - API 키 있으면 실행, 없으면 "테스트 대기" 기록)
      - ## Android 통합 방법 (Firebase AI Logic SDK 사용 시 Kotlin 코드 예시)
      - ## 비용 (무료 티어 한도, 예상 사용량)
      - ## 평가: VIABLE / NOT_VIABLE / NEEDS_API_KEY

4. **NotebookLM MCP 서버 검증** (notebooklm-mcp-test.md):
   프로젝트에 이미 설정된 notebooklm-mcp 서버를 활용하여 검증한다.
   a) MCP 서버 상태 확인: 사용 가능한 도구 목록 (source_add, note_create 등)
   b) 검증 시나리오 설계:
      - 시나리오 1: 새 노트북 생성 -> source_add(text, 전사 텍스트) -> note_create(회의록 생성 요청)
      - 시나리오 2: 기존 노트북에 소스 추가 -> 노트 생성
   c) MCP 서버 제약사항 분석:
      - PC 의존성 (모바일 독립 실행 불가)
      - Google 계정 인증 필요
      - 비공식 경로 (안정성 리스크)
   d) notebooklm-mcp-test.md에 기록:
      - ## MCP 서버 상태 (사용 가능 여부, 도구 목록)
      - ## 테스트 시나리오 (실행 가능한 시나리오)
      - ## 테스트 결과 (MCP 도구 호출 결과 또는 "테스트 대기" 기록)
      - ## 제약사항 (PC 의존성, 안정성)
      - ## Gemini API vs MCP 비교 (모바일 독립성, 안정성, 기능 범위)
      - ## 평가: VIABLE / NOT_VIABLE / LIMITED
