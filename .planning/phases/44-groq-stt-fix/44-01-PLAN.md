# Plan 44-01: GroqSttEngine OkHttp Request 재사용 버그 수정

## Goal

Groq Whisper STT 엔진이 한국어 전사를 정상적으로 수행한다 (BUG-01).

## Context

`GroqSttEngine.kt`에 4가지 버그가 있어 Groq STT가 전혀 동작하지 않는다:

**버그 1 (치명)**: `requestBody`가 재시도 루프 외부에서 한 번만 생성됨.
파일 스트림을 포함하는 `MultipartBody`는 첫 `execute()` 후 스트림이 소진되어 재사용 불가.

**버그 2 (치명)**: OkHttp `Request` 객체도 루프 외부에서 한 번만 생성됨.
동일 `Request`를 재사용하면 두 번째 호출부터 `IllegalStateException` 또는 빈 body 전송.

**버그 3 (중대)**: 429 재시도 경로에서 `response.close()` 미호출.
`response.body?.string()`으로 body를 읽은 뒤 `continue`로 넘어가면 커넥션이 닫히지 않아 OkHttp 커넥션 풀 누수 → MAX_RETRIES 재시도 후 커넥션 고갈로 타임아웃 발생 가능.

**버그 4 (경미)**: `text.isNullOrBlank()` — `String`은 항상 non-null이므로 null 체크 무의미. `isBlank()`로 교체해야 빈 응답 검증이 올바르게 동작.

**추가 개선**: `OkHttpClient`가 `transcribe()` 호출마다 새로 생성됨 (라인 93-97). 커넥션 풀과 스레드 풀을 포함하는 `OkHttpClient`는 `@Singleton` 클래스에서 멤버 변수로 재사용해야 한다.

실제 서비스 흐름 (정상 동작 확인):
- `SttEngineType.GROQ` enum 존재 ✅
- `TranscriptionRepositoryImpl` GROQ 케이스 처리 ✅
- Hilt DI 바인딩 ✅
- SettingsScreen Groq 선택 UI ✅

## Tasks

### Task 1: OkHttpClient를 멤버 변수로 이동 + requestBody/request를 루프 내부로 이동 + response.use{} 적용

**파일**: `app/src/main/java/com/autominuting/data/stt/GroqSttEngine.kt`

전체 `transcribe()` 메서드 내 관련 블록을 아래와 같이 수정한다.

**1-a: OkHttpClient를 companion object 또는 클래스 멤버로 이동**

현재 (라인 93-97, `transcribe()` 내부):
```kotlin
val client = OkHttpClient.Builder()
    .connectTimeout(60, TimeUnit.SECONDS)
    .writeTimeout(5, TimeUnit.MINUTES)
    .readTimeout(5, TimeUnit.MINUTES)
    .build()
```

수정: 클래스 멤버 변수로 선언 (companion object 또는 `private val client` 프로퍼티):
```kotlin
private val client = OkHttpClient.Builder()
    .connectTimeout(60, TimeUnit.SECONDS)
    .writeTimeout(5, TimeUnit.MINUTES)
    .readTimeout(5, TimeUnit.MINUTES)
    .build()
```

**1-b: requestBody + request 생성을 재시도 루프 안으로 이동**

현재 (루프 외부):
```kotlin
val requestBody = MultipartBody.Builder()
    .setType(MultipartBody.FORM)
    .addFormDataPart("file", audioFile.name,
        audioFile.asRequestBody(mimeType.toMediaType()))
    .addFormDataPart("model", MODEL)
    .addFormDataPart("language", "ko")
    .addFormDataPart("response_format", "verbose_json")
    .build()

val request = Request.Builder()
    .url(API_URL)
    .header("Authorization", "Bearer $apiKey")
    .post(requestBody)
    .build()

var lastException: Exception? = null
for (attempt in 1..MAX_RETRIES) {
    try {
        Log.d(TAG, "Groq API 호출 (시도 $attempt/$MAX_RETRIES)")
        val response = client.newCall(request).execute()
        val responseBody = response.body?.string()
```

수정 후:
```kotlin
var lastException: Exception? = null
for (attempt in 1..MAX_RETRIES) {
    try {
        // 매 시도마다 새로운 requestBody와 request 생성 (OkHttp stream 재사용 불가)
        val requestBody = MultipartBody.Builder()
            .setType(MultipartBody.FORM)
            .addFormDataPart(
                "file", audioFile.name,
                audioFile.asRequestBody(mimeType.toMediaType())
            )
            .addFormDataPart("model", MODEL)
            .addFormDataPart("language", "ko")
            .addFormDataPart("response_format", "verbose_json")
            .build()

        val request = Request.Builder()
            .url(API_URL)
            .header("Authorization", "Bearer $apiKey")
            .post(requestBody)
            .build()

        Log.d(TAG, "Groq API 호출 (시도 $attempt/$MAX_RETRIES)")
        val response = client.newCall(request).execute()
        response.use { resp ->
            val responseBody = resp.body?.string()
```

**1-c: response.use{} 블록으로 커넥션 자동 닫힘 보장**

`response.use { resp -> ... }` 블록 내부에서 기존 `responseBody` 처리 로직을 그대로 유지한다. 블록이 끝나면 response가 자동으로 닫힌다.

429 처리 (`resp.code == 429`) 포함 모든 경로에서 response가 닫히게 된다.
